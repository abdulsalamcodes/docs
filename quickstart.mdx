This guide walks you through the core Ancra workflow end-to-end. By the end, you will have created a customer, provisioned a virtual bank account for them, understood how inbound payments are delivered to you, and queried a live account balance — all via the REST API.

## Prerequisites

Before you begin, make sure you have:

- **An Ancra API key** — see [Authentication](/authentication) for how to generate one.
- **Your deployment base URL** — the URL of your Ancra server instance.

Export both as environment variables so you can copy and paste the examples below directly:

```bash
export ANCRA_KEY="ancra_live_xxxxxxxxxxxxxxxxxxxx"
export BASE="https://your-deployment.onrender.com"
```

---

<Steps>
  <Step title="Create a Customer">
    Every virtual account in Ancra belongs to a customer. Start by creating a customer record. The `kyc_tier` field controls the transaction limits applied to the customer's accounts.

    ```bash
    curl -X POST $BASE/customers \
      -H "Authorization: Bearer $ANCRA_KEY" \
      -H "Content-Type: application/json" \
      -d '{"kyc_tier": 1}'
    ```

    Ancra returns the new customer object:

    ```json
    {
      "id": "c8a1b2d3-...",
      "kyc_tier": 1,
      "created_at": "2026-01-01T00:00:00Z"
    }
    ```

    Save the `id` field — you will need it in the next step to provision a virtual account for this customer.
  </Step>

  <Step title="Provision a Virtual Account">
    With your customer ID in hand, request a Dedicated Virtual Account. Pass the customer's display name, email, and BVN so the provider can associate the account correctly.

    ```bash
    curl -X POST $BASE/accounts \
      -H "Authorization: Bearer $ANCRA_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "customer_id": "c8a1b2d3-...",
        "display_name": "Amara Okonkwo",
        "customer_email": "amara@example.com",
        "bvn": "12345678901"
      }'
    ```

    The response contains the provisioned account details:

    ```json
    {
      "account": {
        "id": "a1b2c3d4-...",
        "bank_account_number": "9876543210",
        "bank_account_name": "Amara Okonkwo",
        "status": "active",
        "created_at": "2026-01-01T00:00:00Z"
      }
    }
    ```

    The `bank_account_number` field is the real Nigerian bank account number your customer shares with anyone who wants to pay them. Any NIP transfer sent to that number will be captured by Ancra and delivered to your webhook.
  </Step>

  <Step title="Receive a Payment">
    You do not need to poll for incoming payments. When someone transfers money to `9876543210`, the payment provider fires a `payment_success` event to the webhook URL you have registered with Ancra.

    Ancra's webhook handler:
    1. Verifies the provider's HMAC signature to confirm authenticity.
    2. Credits the customer's double-entry ledger for the received amount.
    3. Responds `200` to the provider to acknowledge receipt.
    4. Delivers the verified event payload to your registered webhook URL.

    Your webhook endpoint receives the event and can act on it immediately — for example, to unlock a feature, send a receipt, or trigger a downstream workflow.

    <Info>
      See [Webhooks Overview](/webhooks/overview) for full details on registering your endpoint, verifying signatures, and handling event payloads.
    </Info>
  </Step>

  <Step title="Check the Balance">
    You can query an account's current ledger balance at any time using the account ID returned in Step 2.

    ```bash
    curl $BASE/accounts/a1b2c3d4-.../balance \
      -H "Authorization: Bearer $ANCRA_KEY"
    ```

    ```json
    {
      "AccountID": "a1b2c3d4-...",
      "Balance": 100000,
      "Currency": "NGN",
      "AsOf": "2026-01-01T00:01:00Z"
    }
    ```

    The `Balance` field is denominated in **kobo**. In this example, `100000` kobo equals ₦1,000. Divide by `100` to convert to Naira for display purposes.
  </Step>
</Steps>
