The Webhooks API includes the public endpoint that receives events from Nomba, and an admin endpoint for inspecting delivery history. The receiver endpoint is called by Nomba automatically whenever a transaction event occurs — you do not call it yourself. The admin endpoint lets you audit every inbound delivery attempt for debugging and reconciliation purposes.

---

## POST /webhooks/nomba

### Nomba Webhook Receiver

This is the public-facing endpoint Nomba calls to notify Ancra of transaction events such as inbound payments and transfer outcomes.

```http
POST /webhooks/nomba
```

<Note>
  This endpoint is called **by Nomba**, not by you. You only need to interact with it indirectly — by configuring Nomba to point at your Ancra deployment's `/webhooks/nomba` URL. See the [Webhook Overview](/webhooks/overview) for setup instructions.
</Note>

#### Security

Although the endpoint is publicly reachable, every incoming request is authenticated using **HMAC-SHA256 signature verification**. Ancra computes the expected signature from the raw request body and your shared Nomba webhook secret, then compares it against the signature Nomba includes in the request headers.

- Requests with a **valid signature** are processed normally.
- Requests with an **invalid or missing signature** are rejected immediately with `401 Unauthorized`.

See the [Webhook Signature Verification](/webhooks/verification) guide for a detailed breakdown of how verification works.

#### Responses

| HTTP Status | Body | Meaning |
|-------------|------|---------|
| `200 OK` | `{"status": "ok"}` | The event was processed successfully. |
| `200 OK` | `{"status": "ok"}` | A duplicate event was received — Ancra ignores it idempotently so Nomba does not retry. |
| `200 OK` | `{"status": "ignored"}` | The event type is not handled by Ancra (e.g. an event category you have not subscribed to). |
| `401 Unauthorized` | — | The HMAC-SHA256 signature is invalid or the `X-Nomba-Signature` header is absent. |

```json
{ "status": "ok" }
```

```json
{ "status": "ignored" }
```

<Warning>
  Never expose your Nomba webhook secret in client-side code or version control. If the secret is compromised, rotate it immediately in the Nomba dashboard and update your Ancra environment variables to match.
</Warning>

## GET /admin/webhooks

### List Webhook Deliveries

Returns a paginated list of all inbound webhook delivery attempts recorded by Ancra. Use this endpoint to audit event delivery, investigate missed events, and confirm that your Nomba configuration is correctly forwarding events.

```http
GET /admin/webhooks
Admin-Secret: <secret>
```

<Note>
  This endpoint requires the `Admin-Secret` header. The value must match the `ADMIN_SECRET` environment variable configured in your Ancra deployment. It is **not** a Bearer token — pass it as a raw header value.
</Note>

#### Query Parameters

<ParamField query="limit" type="integer">
  Maximum number of records to return per page. Defaults to `20`.
</ParamField>

<ParamField query="offset" type="integer">
  Number of records to skip before returning results. Use together with `limit` to paginate through large delivery histories. Defaults to `0`.
</ParamField>

#### Response `200 OK`

<ResponseField name="webhooks" type="array">
  An array of webhook delivery attempt objects, ordered most-recent first.

  <Expandable title="webhook delivery object fields">
    <ResponseField name="id" type="string">
      Unique identifier for this delivery attempt (UUID format).
    </ResponseField>

    <ResponseField name="event_type" type="string">
      The Nomba event type that triggered this delivery (e.g. `"payment_success"`, `"payment_reversal"`).
    </ResponseField>

    <ResponseField name="status" type="string">
      Delivery outcome. Common values: `"delivered"`, `"failed"`, `"pending"`.
    </ResponseField>

    <ResponseField name="attempts" type="integer">
      Total number of delivery attempts made for this event, including retries.
    </ResponseField>

    <ResponseField name="created_at" type="string">
      ISO 8601 timestamp of when the delivery record was first created.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="limit" type="integer">
  The `limit` value used for this page of results.
</ResponseField>

<ResponseField name="offset" type="integer">
  The `offset` value used for this page of results.
</ResponseField>

```json
{
  "webhooks": [
    {
      "id": "w1b2c3d4-e5f6-7890-abcd-ef1234567890",
      "event_type": "payment_success",
      "status": "delivered",
      "attempts": 1,
      "created_at": "2026-01-01T10:00:00Z"
    }
  ],
  "limit": 20,
  "offset": 0
}
```

<Tip>
  If you are investigating a missed payment, filter the list for records where `status` is `"failed"` and `event_type` is `"payment_success"`. Cross-reference those records with your `GET /reconciliation` report to confirm whether the discrepancy originated from a delivery failure.
</Tip>
